여기에 **ui.rules의 effect.lookup.filters를 평가해서 lookup page filters로 세팅**하면 된다.

### ✅ 수정 포인트 1: openLookupModal에서 “동적 filters” 세팅

`/src/generic/component/GenericDetail.js`의 `openLookupModal`을 아래처럼 확장하면 됨.

핵심은:

* 해당 field의 `ui.rules`를 찾아서
* `when`이 true인 rule들의 `effect.lookup.filters`를 평가해서
* lookup page의 `filters`로 넣고 fetch

아래는 **붙여넣기 가능한 형태**로 작성했어(최소 침습).

```js
// GenericDetailV3 내부에 추가 (openLookupModal 위쪽이 가장 자연스러움)
const uiRulesByField = useMemo(() => {
  const m = {};
  (definitions || []).forEach((def) => {
    const colInfo = def.join && def.join.originColInfo ? def.join.originColInfo : null;
    const rules = colInfo && colInfo.ui && Array.isArray(colInfo.ui.rules) ? colInfo.ui.rules : null;
    if (rules && rules.length) {
      m[def.formFieldName] = rules;
    }
  });
  return m;
}, [definitions]);

const buildRuleCtx = useCallback(() => {
  const wf = workflowRuntime && typeof workflowRuntime === 'object'
    ? (workflowRuntime.instance || { status: (workflowRuntime.lock ? 'RUNNING' : undefined) })
    : null;

  return {
    form: (watchedFormValues && typeof watchedFormValues === 'object')
      ? watchedFormValues
      : ((detail && typeof detail === 'object') ? detail : {}),
    workflow: (wf && typeof wf === 'object') ? wf : {},
  };
}, [watchedFormValues, detail, workflowRuntime]);

const modeScope = isViewMode ? 'view' : ((isInsertMode || isUpdateMode) ? 'edit' : 'all');

const extractLookupFiltersFromRules = useCallback((fieldKey) => {
  const rules = uiRulesByField[fieldKey];
  if (!Array.isArray(rules) || !rules.length) {
    return { filters: [], resetOn: [] };
  }

  const ctx = buildRuleCtx();
  const outFilters = [];
  const resetOnSet = new Set();

  for (let i = 0; i < rules.length; i += 1) {
    const r = rules[i];
    if (!r || r.enabled === false) continue;

    const scopeOk = !(r.scope && modeScope && String(r.scope) !== String(modeScope) && String(r.scope) !== 'all');
    if (!scopeOk) continue;

    const whenOk = r.when ? !!evalJsonLogic(r.when, ctx) : true;
    if (!whenOk) continue;

    const eff = r.effect && typeof r.effect === 'object' ? r.effect : {};
    const lk = eff.lookup && typeof eff.lookup === 'object' ? eff.lookup : null;
    if (!lk) continue;

    if (Array.isArray(lk.resetOn)) {
      lk.resetOn.forEach((x) => resetOnSet.add(String(x)));
    }

    if (Array.isArray(lk.filters)) {
      lk.filters.forEach((f) => {
        if (!f || typeof f !== 'object') return;
        const dbColumnName = f.dbColumnName || f.col;
        if (!dbColumnName) return;

        const v = (typeof f.value === 'object')
          ? evalJsonLogic(f.value, ctx)
          : f.value;

        if (v === undefined || v === null) return;
        if (typeof v === 'string' && v.trim() === '') return;
        if (Array.isArray(v) && v.length === 0) return;

        outFilters.push({ dbColumnName, value: v });
      });
    }
  }

  return { filters: outFilters, resetOn: Array.from(resetOnSet) };
}, [uiRulesByField, buildRuleCtx, modeScope]);

// 기존 openLookupModal 수정
const openLookupModal = useCallback(
  (fieldKey) => {
    const info = parentLookupMap[fieldKey];
    if (!info) return;

    const rootHostPageId = page && page.parentPageId ? page.parentPageId : null;
    if (!rootHostPageId) return;

    const lookupPageId = `${pageId}__lookup__${fieldKey}`;

    // ✅ 여기: ui.rules 기반 동적 필터 추출
    const { filters } = extractLookupFiltersFromRules(fieldKey);

    const { actions: lookupActions } = ensurePageInjected(lookupPageId, {
      domain,
      entityId: info.parentEntityId,
      entityNm: info.parentEntityNm,
      openType: 'modal',
      parentPageId: rootHostPageId,
      isOpen: true,
      uiType: 'list',
      mode: 'select',
      filters, // ✅ 초기 filters 주입
      lookup: {
        targetPageId: pageId,
        targetField: fieldKey,
        valueKey: info.valueKey || 'id',
        labelKey: info.labelKey || 'id',
        labelField: `${fieldKey}__label`,
      },
    });

    dispatch(lookupActions.open());

    // ✅ page가 이미 주입된 상태일 수 있으니 open 직전에 setFilters도 해줌
    dispatch(lookupActions.setFilters(filters));
    dispatch(lookupActions.fetchListRequested({ pageId: lookupPageId }));
  },
  [parentLookupMap, page, pageId, dispatch, domain, extractLookupFiltersFromRules]
);
```

### ✅ 수정 포인트 2: resetOn 구현(부모가 바뀌면 자식 자동 초기화)

너 GenericDetail은 FK 표시값을 `detail`에서 그리기 때문에,
폼만 지우면 화면이 안 바뀌는 문제가 있어.
그래서 **actions.patchDetail로 지우는 게 맞아.**

```js
const dependentResetMap = useMemo(() => {
  const m = {};
  Object.keys(uiRulesByField || {}).forEach((fieldKey) => {
    const rules = uiRulesByField[fieldKey];
    if (!Array.isArray(rules)) return;

    rules.forEach((r) => {
      const lk = r && r.effect && r.effect.lookup ? r.effect.lookup : null;
      if (lk && Array.isArray(lk.resetOn) && lk.resetOn.length) {
        m[fieldKey] = Array.from(new Set([...(m[fieldKey] || []), ...lk.resetOn.map(String)]));
      }
    });
  });
  return m;
}, [uiRulesByField]);

const prevFormRef = useRef(null);

useEffect(() => {
  const cur = watchedFormValues && typeof watchedFormValues === 'object' ? watchedFormValues : {};
  const prev = prevFormRef.current;
  prevFormRef.current = cur;
  if (!prev) return;

  Object.entries(dependentResetMap || {}).forEach(([childField, parents]) => {
    const changed = (parents || []).some((p) => prev[p] !== cur[p]);
    if (!changed) return;

    const childLabel = `${childField}__label`;
    const hasChild =
      (cur[childField] !== undefined && cur[childField] !== null && cur[childField] !== '') ||
      (cur[childLabel] !== undefined && cur[childLabel] !== null && cur[childLabel] !== '');

    if (hasChild && actions && typeof actions.patchDetail === 'function') {
      dispatch(actions.patchDetail({ [childField]: null, [childLabel]: null }));
    }
  });
}, [watchedFormValues, dependentResetMap, actions, dispatch]);
```

# 왜 dependent select가 반드시 필요한가?

`travel_preference`에는 2개의 FK가 있지:

* `pref_key_id` → `travel_pref_key.id`
* `value_option_id` → `travel_pref_value_option.id` (nullable)

그리고 `travel_pref_value_option`은 본질적으로 **특정 pref_key에 종속**돼:

* `travel_pref_value_option.pref_key_id = travel_pref_key.id`

즉, **(Key를 먼저 고르고) 그 Key의 옵션만 선택해야 정상**이야.

만약 UI에서 이걸 커버하지 않으면:

1. 사용자가 `pref_key_id=origin_airport` 골라놓고
2. `value_option_id`에서 `seat_class` 옵션을 골라도
3. DB에는 “겉보기로는 FK가 다 맞아서” 저장될 수 있음 (각각 존재하는 id라서)

→ 이건 실무에서 치명적인 데이터 오류야.

그래서 UX/검증/데이터 품질을 위해 **Key 선택 → Option 필터링**은 사실상 필수야.

---

# 목표 동작(정상 UX)

1. `pref_key_id`를 먼저 선택
2. `value_option_id` 선택 UI를 열면

   * 옵션 목록은 자동으로
     `travel_pref_value_option.pref_key_id = (선택된 pref_key_id)`
     로 필터링
3. `pref_key_id`가 바뀌면 `value_option_id`는 자동 초기화(null)

---

# 구현 방법: 메타 기반으로 “한 번만” 만들기

핵심은 **travel만을 위한 코드가 아니라**
**모든 도메인에서 재사용 가능한 dependent lookup 규칙**으로 만들자는 거야.

## A안: 기존 `ui_field_rule`에 “dependent lookup” 규칙을 추가 (추천)

너 시스템은 이미 `ui_field_rule`로 UI 제어를 하고 있으니까,
테이블 추가 없이 `effect_json`에 규칙을 하나 더 넣는 방식이 가장 가볍다.

### Rule 1) pref_key_id 없으면 value_option_id 비활성화

* 대상: `travel_preference.value_option_id`

**when_json**

```json
{ "!": { "var": "form.prefKeyId" } }
```

**effect_json**

```json
{ "disabled": true, "message": "Preference Key를 먼저 선택하세요." }
```

### Rule 2) pref_key_id 있으면 lookup에 필터 주입 + resetOn

**when_json**

```json
{ "var": "form.prefKeyId" }
```

**effect_json**

```json
{
  "lookup": {
    "filters": [
      { "dbColumnName": "pref_key_id", "value": { "var": "form.prefKeyId" } }
    ],
    "resetOn": ["prefKeyId"]
  }
}
```

* `filters` : lookup 대상 테이블(=travel_pref_value_option)의 where 조건
* `resetOn` : 부모가 바뀌면 자식을 자동 초기화하는 힌트

> 이 룰은 travel 전용이 아니라 “부모 선택에 종속되는 모든 select”에 재사용 가능해.

# 프론트 구현(최소 변경 포인트)

너 genericReact의 select는 보통 **lookup modal(참조 테이블 목록 페이지)**를 재사용하잖아.
그러면 dependent select도 결국 **“lookup modal을 열 때 filters를 추가”**하면 끝이야.

## 1) lookup modal open 시점에 rule 평가

`GenericDetail`에서 “value_option_id lookup 열기” 이벤트가 발생하면:

1. 해당 field(`value_option_id`)의 `ui_field_rule`를 읽고
2. `effect_json.lookup.filters`가 있으면
3. `{ "var": "form.prefKeyId" }`를 현재 폼 값으로 평가해서
4. lookup modal list 조회에 filters로 전달

예시로 실제로 전달되는 filters는 이런 모양이 돼:

```json
[
  { "dbColumnName": "pref_key_id", "value": 206 }
]
```

## 2) pref_key_id가 바뀌면 value_option_id 초기화

* `resetOn: ["prefKeyId"]`가 정의돼 있으니,
* 폼 값 watch 해서 `prefKeyId` 변경 시

  * `valueOptionId = null`
  * (있다면) `valueOptionId__label`도 null

이 로직도 “rules 기반”으로 만들면 다른 dependent select에도 자동 적용돼.

# 백엔드 구현(최소 변경 포인트)

프론트에서 filters를 보내면, 백엔드는 그냥 이를 WHERE로 바꿔주면 된다.

이미 너 시스템이 **filter 기반 목록 조회**(GenericFilter 등)를 지원한다면:

* `GET /ref/...` 같은 별도 API를 만들 필요 없이
* 기존 list command에 `filters`만 넣어주면 됨

예:

* lookup 대상이 `travel_pref_value_option`이면
* filters에 `pref_key_id=206`을 포함해서 전달

→ 서버는 자연스럽게:

```sql
WHERE pref_key_id = 206
```

로 목록을 줄여준다.

# 데이터 무결성(중요): UI가 완벽하지 않아도 DB에서 막기

UI만 믿으면 나중에 API 호출/버그/직접 insert로 데이터가 깨질 수 있어.
그래서 DB 레벨에서 “Key와 Option이 반드시 매칭”되게 제약을 거는 게 좋아.

## Composite FK(추천)

`travel_preference(pref_key_id, value_option_id)`가
`travel_pref_value_option(pref_key_id, id)`를 참조하게 만든다.

(MySQL 예시)

```sql
ALTER TABLE travel_pref_value_option
  ADD UNIQUE KEY uq_pref_key_id_id (pref_key_id, id);

ALTER TABLE travel_preference
  ADD CONSTRAINT fk_pref_option_matches_key
  FOREIGN KEY (pref_key_id, value_option_id)
  REFERENCES travel_pref_value_option (pref_key_id, id);
```

이러면 “존재하는 option id”라도 **다른 key의 option**이면 저장 자체가 실패한다.

---

# 정리: 네 프로젝트에 맞는 “최소” 구현 플랜

1. **메타(rule) 추가**

   * `value_option_id`는 `pref_key_id`에 종속
   * lookup.filters + resetOn 정의

2. **프론트 1회 일반화**

   * lookup modal open 시 rules를 읽어 filters를 주입
   * resetOn 감지해 child 값 자동 초기화

3. **DB 무결성**

   * composite FK로 잘못된 조합 원천 차단






# 예시

## 1) 왜 dependent select가 “필요”한가 (너 케이스에서 더 중요)

`travel_preference`에서

* `pref_key_id` (예: travel_purpose, origin_airport, seat_class …)
* `value_option_id` (enum일 때만 선택)

이 구조는 **value_option_id가 “항상 특정 pref_key_id에 종속”**이야.

즉, `value_option_id`는 아무 옵션이나 고르면 안 되고:

> `travel_pref_value_option.pref_key_id == travel_preference.pref_key_id`

가 반드시 만족되어야 정상 데이터.

이 필터링이 UI에 없으면:

* 사용자가 `pref_key_id = origin_airport` 선택하고
* `value_option_id`를 `seat_class` 옵션으로 선택해도
* DB FK는 각각 존재하니까 “겉보기엔 저장”될 수 있음
  → **데이터 오염**이 발생함.

그래서 **Key 선택 → Option 필터링**은 UX가 아니라 **데이터 품질/무결성** 기능이야.

---

## 2) 너 시스템에서 가장 “깔끔한” 구현 방식

너 backend(`/api/{domain}/schema/full`)는 이미 **ui_field_rule**을 읽어서 schema에 `col_meta.ui.rules`로 내려주고 있어. (llm_backend `domain.py`에서 `_load_ui_field_rules`로 내려줌)

그리고 front(GenericDetail.js)는 이미:

* `ui.rules`를 읽어서
* `when_json`을 `evalJsonLogic()`로 평가하고
* `effect.disabled`를 적용해서 disable 처리까지 하고 있음.

✅ 즉, 너는 “새로운 시스템”을 만드는 게 아니라
